## 功能概述

### 解决的问题

**问题：** 用户直接对音箱说"播放周杰伦的晴天"时，指令由小爱云服务处理，不会经过你的本地服务，无法使用你的音乐源。

**解决：** 通过持续监听音箱的对话记录，自动拦截播放指令，调用你的音乐服务获取 URL 并播放。

### 参考项目

本功能参考 [xiaomusic](https://github.com/hanxi/xiaomusic) 项目实现。

## 工作原理

### 整体架构

```
┌──────────────┐
│   用户说话   │
│ "播放晴天"   │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 小爱音箱云端 │
│  记录对话    │
└──────┬───────┘
       │
       ▼ (每2秒轮询)
┌──────────────────┐
│ ConversationPoller │
│   检测新对话      │
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ CommandHandler   │
│  解析播放指令    │
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│  音乐搜索服务    │
│  (MUSIC_API)     │
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ play_music 接口  │
│  获取音乐 URL    │
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│  音箱播放音乐    │
└──────────────────┘
```

### 核心组件

#### 1. ConversationPoller（对话轮询器）

**职责：** 持续监听所有音箱的对话记录

**工作流程：**
```python
while True:
    for device in devices:
        # 获取对话记录
        conversations = await client.get_latest_ask(device_id)
        
        # 处理新对话
        for conv in conversations:
            if conv["time"] > last_timestamp:
                # 触发命令处理
                await command_callback(device_id, conv["query"])
    
    await asyncio.sleep(poll_interval)
```

#### 2. CommandHandler（命令处理器）

**职责：** 解析指令并执行播放流程

**支持的指令格式：**
- `播放{歌名}` - 例如："播放晴天"
- `播放{歌手}的{歌名}` - 例如："播放周杰伦的晴天"
- `播放歌曲{歌名}` - 例如："播放歌曲晴天"
- `打开{歌名}` - 例如："打开晴天"

**过滤规则：**

不会拦截包含以下关键词的指令：
- 音量、暂停、继续、停止、下一首、上一首

**处理流程：**
```python
async def handle_command(device_id, query):
    # 1. 解析指令
    play_info = _parse_play_command(query)
    if not play_info:
        return  # 不是播放指令
    
    # 2. 搜索音乐
    results = await _search_music(play_info["query"])
    
    # 3. 同步播放列表
    await _sync_playlist(device_id, results["songs"])
    
    # 4. 播放第一首
    await _play_song(device_id, 0)
```

#### 3. 对话 API

**API 地址：**
```
https://userprofile.mina.mi.com/device_profile/v2/conversation
```

**请求参数：**
- `source=dialogu`
- `hardware={设备型号}`
- `timestamp={当前时间戳}`
- `limit=2`

**认证方式：**
```
Cookie: deviceId={设备ID}; serviceToken={服务令牌}
```

**响应示例：**
```json
{
  "code": 0,
  "data": "{\"records\":[{\"time\":1710825600000,\"query\":\"播放周杰伦的晴天\",\"answers\":[{\"tts\":{\"text\":\"好的，正在为您播放\"}}]}]}"
}
```

## 故障排查

### 问题 1：服务启动但没有日志

**症状：**
```
# 没有看到这行日志
INFO xiaoai_media.conversation — 对话轮询器已启动
```

**检查：**
1. 查看 `.env` 文件：
   ```bash
   cat .env | grep ENABLE_CONVERSATION_POLLING
   ```
   应该是 `true`

2. 查看启动日志是否有错误

**解决：**
```bash
# 确保配置正确
echo "ENABLE_CONVERSATION_POLLING=true" >> .env
```

### 问题 2：报错 "get_latest_ask failed"

**症状：**
```
ERROR — MiService: get_latest_ask failed: ...
```

**可能原因：**
- 小米账号认证失败
- serviceToken 过期
- 网络连接问题

**解决：**
1. 检查账号配置：
   ```bash
   curl http://localhost:8000/api/devices
   ```
   如果返回设备列表，说明认证成功

2. 重新获取 passToken：
   ```bash
   cd backend
   python3 -c "from miservice import MiAccount; import asyncio; from aiohttp import ClientSession; async def test(): async with ClientSession() as s: acc = MiAccount(s, 'user', 'pass'); await acc.login(); print(acc.service_token); asyncio.run(test())"
   ```

### 问题 3：检测到对话但没有播放

**症状：**
```
INFO — 检测到新对话: 播放周杰伦的晴天
# 但没有后续的播放日志
```

**检查：**
1. 音乐服务是否运行：
   ```bash
   curl http://localhost:5050/api/v3/search -X POST \
     -H "Content-Type: application/json" \
     -d '{"platform":"tx","query":"晴天","page":1,"limit":20}'
   ```

2. 查看详细错误日志

**解决：**
- 确保音乐下载服务正在运行
- 检查 `MUSIC_API_BASE_URL` 配置

### 问题 4：播放的歌曲不对

**症状：** 说"播放晴天"，但播放的不是周杰伦的版本

**原因：** 搜索结果的第一首可能是其他歌手的

**解决：**
- 说话时包含歌手名："播放周杰伦的晴天"
- 或者在 Web 界面手动选择播放

### 问题 5：响应太慢

**症状：** 说完话后等很久才开始播放

**原因：** 轮询间隔太长

**解决：**
```bash
# 缩短轮询间隔到 1 秒
CONVERSATION_POLL_INTERVAL=1.0
```

**注意：** 不建议设置太短（< 0.5秒），会增加 API 请求频率

### 时序图

```
用户 -> 音箱: "小爱同学，播放晴天"
音箱 -> 云端: 记录对话
云端 -> 云端: 存储对话记录

[2秒后]
轮询器 -> 云端: GET /conversation (获取对话)
云端 -> 轮询器: 返回对话列表
轮询器 -> 轮询器: 检测到新对话 "播放晴天"
轮询器 -> 命令处理器: handle_command("播放晴天")
命令处理器 -> 命令处理器: 解析为播放指令
命令处理器 -> 音乐服务: POST /search (搜索"晴天")
音乐服务 -> 命令处理器: 返回歌曲列表
命令处理器 -> 命令处理器: 同步播放列表
命令处理器 -> 音乐服务: POST /play (获取URL)
音乐服务 -> 命令处理器: 返回播放URL
命令处理器 -> 音箱: play_url(url)
音箱 -> 音箱: 开始播放
```

## 性能考虑

### 资源消耗

| 资源 | 影响 | 说明 |
|------|------|------|
| CPU | 极低 | 轮询任务很轻量 |
| 内存 | 极低 | 只存储时间戳 |
| 网络 | 低 | 每个轮询周期一次请求/设备 |
| API 配额 | 低 | 小米 API 无明确限制 |

### 优化建议

1. **单设备场景**
   - 可以缩短轮询间隔到 1 秒

2. **多设备场景**
   - 建议保持 2 秒间隔
   - 或者只监听常用设备

3. **省电模式**
   - 增加轮询间隔到 5 秒
   - 或者禁用对话监听，只用 Web 控制

## 常见问题 FAQ

**Q: 为什么需要轮询？不能实时推送吗？**

A: 小米没有提供实时推送 API，只能通过轮询对话记录来检测新指令。xiaomusic 也是用这种方式。

**Q: 会不会漏掉指令？**

A: 理论上不会。只要轮询间隔不是太长（建议 ≤ 5秒），都能检测到。系统会记录时间戳，确保不遗漏。

**Q: 可以只监听特定设备吗？**

A: 目前会监听所有设备。如果需要只监听特定设备，可以修改 `conversation.py` 中的 `_poll_conversations()` 方法添加过滤逻辑。

**Q: 对话监听会影响正常使用吗？**

A: 不会。对话监听只是读取对话记录，不会干扰音箱的正常功能。

**Q: 如果音乐服务挂了会怎样？**

A: 播放会失败，但不影响其他功能。日志会记录错误信息。
